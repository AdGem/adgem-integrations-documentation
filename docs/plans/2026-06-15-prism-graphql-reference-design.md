# Prism (GraphQL) Reference — Design

**Date:** 2026-06-15
**Context:** WS3 of the docs-v2 launch plan. The Offer API (REST) reference is done (PR #44). This designs the **Prism (GraphQL)** reference — the other half of WS3. Generated via `@graphql-markdown/docusaurus` (`npm run graphql-to-doc`) from the live Prism schema; config already repointed (`baseURL: reference/prism`, homepage `static/prism.md`, `PRISM_SCHEMA_URL`/`PRISM_ACCESS_TOKEN`).

## Goal & scope

Build the Prism reference using GraphQL's conventional shape (Operations + Types) with hand-written conceptual pages around it. **In scope:** everything except the interactive playground. **Out of scope:** the playground page (`reference/prism/playground`) — the GraphiQL component and config stay for a later playground task; the page is simply excluded from this work.

## IA / sidebar structure (decided: standard GraphQL layout)

```
Reference
  Prism (GraphQL)              [landing: overview]
    Authentication · Tokens · Errors · Versioning · Recipes · FAQ   ← hand-written conceptual
    Operations                 ← auto-generated
      Queries        (offers, links, …)
      Directives     (include, skip, …)
      [Mutations / Subscriptions only if the schema exposes them]
    Types                      ← auto-generated
      Objects        (Offer, OfferGoal, OfferCreatives, …)
      Inputs         (RetrievalFilters, …)
      Scalars        (Boolean, Int, …)
      [Enums / Interfaces / Unions only if present]
   (no Playground)
```

Rationale: this is the shape GraphQL consumers expect, and `@graphql-markdown` emits it folder-based with `_category_.yml` files, so the schema layout is auto-driven (low maintenance). The conceptual pages sit above it. (This intentionally differs from the hand-curated Offer API REST layout — GraphQL convention over cross-API symmetry.)

## Generated vs. hand-written

- **Generated** (`graphql-to-doc`): the `operations/` and `types/` trees under `reference/prism/`, with `_category_.yml` driving labels/order; Prism-branded via the `static/prism.md` homepage template. These **replace** the skeleton's placeholder pages `reference/prism/schema/{queries,mutations,types,enums,inputs}.md`, which do not match the generator's actual output.
- **Hand-written conceptual** (placeholder pages now; real content is WS2): `overview` (landing), `authentication`, `tokens`, `errors`, `versioning`, `recipes`, `faq`.
  - Migration sources (WS2): `authentication`, `tokens`, and `faq` can migrate from the old Targeted API pages on `main` @ `653b3f3` (`docs/integrations/targeted-api/`); the old `offers-endpoint` content folds into `overview`/`recipes`. `overview`, `errors`, `versioning` are fresh authoring.
  - `faq` is a **new page slot** added per the decision (the old Targeted API had a FAQ; the remap table had flagged it as having no home).

## Sidebar mechanics

In `sidebars.ts`, the `Prism (GraphQL)` category (link → `reference/prism/overview`) lists:
1. the conceptual pages explicitly (`authentication`, `tokens`, `errors`, `versioning`, `recipes`, `faq`), then
2. two `autogenerated` items: `{ type: "autogenerated", dirName: "reference/prism/operations" }` and `{ type: "autogenerated", dirName: "reference/prism/types" }`.

This keeps conceptual pages curated while letting the generated folders + their `_category_.yml` drive the Operations/Types layout. The `playground` page is omitted from the sidebar.

## Implementation gate — token

`npm run graphql-to-doc` introspects the live Prism schema and requires a short-lived Cognito **access token** in `.env` (`PRISM_ACCESS_TOKEN`) plus network reachability to `PRISM_SCHEMA_URL`. The structure/wiring/conceptual-page work is token-free and can be done first; the generated `operations/`+`types/` trees require the token. At implementation time, either the token is provided / the command is run by the maintainer, or the agent attempts it (sandbox reachability to the endpoint is uncertain). If generation can't run, the branch lands the structure + conceptual pages and the generated trees are filled in once a token is available.

## Acceptance criteria

- `npm run build` passes (no broken links / no orphaned ApiItem-style pages).
- Sidebar shows Prism with conceptual pages + auto-generated Operations and Types in the standard GraphQL layout; no `schema/*` placeholder remnants; no Playground entry.
- Generated schema pages are Prism-branded (no "Targeted API" strings).
- Overview is the Prism landing; FAQ page exists.
