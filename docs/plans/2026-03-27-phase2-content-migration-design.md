## Conversion Rules

### Markdown to MDX

| MkDocs (pymdownx) | Docusaurus | Notes |
|---|---|---|
| `=== "Tab Name"` (tabbed blocks) | `<Tabs>` / `<TabItem>` components | Import from `@docusaurus/theme-common` |
| `!!! note "Title"` (admonitions) | `:::note[Title]` | Same for `tip`, `warning`, `info`, `danger` |
| `??? note "Title"` (collapsible) | `<details><summary>Title</summary>...</details>` | Or `:::note` if collapsibility isn't important |
| `{: .class }` (attr_list) | Remove or convert to JSX | Rarely needed |
| `{%- include "file.md" -%}` | `import Partial from './_partials/file.mdx'` then `<Partial />` | MDX import syntax |
| Raw HTML `<img>` tags | Keep as-is or convert to `![alt](path)` | Both work in MDX |
| Emoji shortcodes (`:emoji:`) | Remove or use Unicode directly | pymdownx emoji extension not available |

### Frontmatter

Every migrated `.mdx` file gets standard Docusaurus frontmatter:

```yaml
---
sidebar_position: 1
title: "Page Title"
description: "Brief description for SEO and link previews"
---
```

`sidebar_position` determines ordering within autogenerated sidebar categories. Use the order from the MkDocs nav as a starting point.

### Images

- Copy relevant images from the MkDocs `img/` subdirectories into a local `img/` directory within the target section (e.g., `docs/webhooks/img/`)
- Update image references to relative paths: `./img/filename.png`
- Don't optimize, resize, or replace images — port as-is
- Only copy images that are actually referenced by the migrated content

### URL Cleanup

- Replace hardcoded `https://docs.adgem.com/publisher-support/...` URLs with relative Docusaurus doc paths (e.g., `/docs/webhooks/offer-event-webhooks`)
- Replace hardcoded `https://docs.adgem.com/player-support/...` URLs with a placeholder comment: `<!-- TODO: Update to WordPress player support URL -->`
- Keep external URLs as-is: `dashboard.adgem.com`, `paypal.com`, `github.com`, etc.
- Cross-references between Phase 2 sections should use relative paths even if the target section hasn't been migrated yet — the stubs exist at the right paths

### Existing Stubs

Each target directory already has a stub `index.mdx` with placeholder content. Replace these with the real migrated content. Don't create a separate file — overwrite the stub.

| Partial | Use instead of... |

### File Naming

- Use kebab-case for all new `.mdx` files
- Match the MkDocs filename where reasonable (e.g., `offer-event-webhooks.md` becomes `offer-event-webhooks.mdx`)
- Section index files stay as `index.mdx`

## PR Structure

Each ticket gets its own branch from `origin/main` and its own PR. Branch naming: `ronco/agpi-XXXX-<section-name>`.

Every plan follows the same task structure:

1. Create feature branch from `origin/main`
2. Copy images (if any)
3. Migrate content files (convert markdown, add frontmatter, update paths)
4. Replace stub `index.mdx` with real content
5. Build verification (`npm run build`)
6. Visual verification (Playwright screenshot of the migrated section)
7. Commit and PR
